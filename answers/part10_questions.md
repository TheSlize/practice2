## Вопрос 1

Что выведет этот код? Объясните порядок выполнения блоков.

```java
class Demo {
    static { System.out.print("A "); }
    { System.out.print("B "); }
    Demo() { System.out.print("C "); }
    public static void main(String[] args) {
        System.out.print("D ");
        new Demo();
        System.out.print("E ");
        new Demo();
    }
}
```

**Предсказанный вывод:**

```
A D B C E B C 
```

**Объяснение:**
1. При запуске программы класс `Demo` загружается в память. В этот момент (до запуска `main`) единожды выполняется **статический блок инициализации** -> печатается `A `.
2. Начинает работу метод `main` -> печатается `D `.
3. При вызове первого `new Demo()` сначала выполняется **нестатический (логический) блок инициализации** -> `B `, а только затем **конструктор** -> `C `.
4. Продолжается выполнение `main` -> печатается `E `.
5. При вызове второго `new Demo()` статический блок уже не вызывается. Снова отрабатывают нестатический блок и конструктор -> `B C `.

## Вопрос 2

Класс `Child` в пакете `b`, класс `Parent` в пакете `a`. `Child extends Parent`. Какие члены `Parent` доступны в `Child`? А какие доступны в другом классе пакета `b`, который НЕ наследует `Parent`?

**Ваш ответ:**
1. **В классе `Child` (наследник в другом пакете)** доступны члены `Parent` с модификаторами `public` и `protected`. Члены с модификаторами `private` и `package-private` (по умолчанию) — недоступны.
2. **В другом классе пакета `b` (не наследник)** будут доступны только `public` члены класса `Parent`. Модификатор `protected` здесь уже не сработает, так как нет наследования, а пакеты разные.

## Вопрос 3

Можно ли создать sealed-класс, у которого все наследники — record? Какое преимущество это даёт при использовании в switch?

**Ваш ответ:**

Ремарка: `record` не может наследовать другие классы (он уже неявно наследует `java.lang.Record`), но он **может реализовывать `sealed interface`**. То есть мы можем создать запечатанный *интерфейс*, реализациями которого будут только `record`.

**Преимущество в switch:** Это дает проверку на **исчерпываемость (exhaustiveness)**. Компилятор точно знает все возможные варианты (так как интерфейс sealed). Если в `switch` перечислить все `record`-наследники, компилятор не потребует писать ветку `default`.

## Вопрос 4

Что выведет этот код? Объясните, что такое String Pool и как работает `intern()`.

```java
String a = "Hello";
String b = new String("Hello");
String c = b.intern();
System.out.println(a == b);
System.out.println(a == c);
System.out.println(b == c);
```

**Предсказанный вывод:**

```
false
true
false
```

**Объяснение:**
- `String Pool` (Пул строк) — это специальная область в памяти (в Heap), где кэшируются строковые литералы для экономии памяти.
- `a` указывает на строку в пуле.
- `b` создается через `new`, поэтому принудительно создается **новый объект** в куче (Heap), вне пула. Поэтому `a == b` дает `false`.
- Метод `intern()` берет строку `b`, ищет такое же значение ("Hello") в String Pool. Так как она там уже есть (создана переменной `a`), `intern()` возвращает ссылку на объект из пула.
- Теперь `c` и `a` ссылаются на один и тот же объект в пуле, поэтому `a == c` это `true`, а `b == c` — `false`.

## Вопрос 5

Почему этот код не компилируется? Как исправить?

```java
interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }
class C implements A, B { }
```

**Ваш ответ:**

Код не компилируется из-за **diamond problem-а**. Класс `C` наследует две одинаковые дефолтные реализации метода `hello()` от интерфейсов `A` и `B`, и компилятор не знает, какую из них выбрать.

**Как исправить:** Класс `C` обязан переопределить этот метод. Внутри переопределенного метода можно написать свою логику, либо явно вызвать метод одного из интерфейсов:
```java
class C implements A, B {
    @Override
    public void hello() {
        A.super.hello(); // Явный выбор реализации из интерфейса A
    }
}
```

## Вопрос 6

В чём разница между `StringBuilder` и `StringBuffer`? Приведите конкретный сценарий, когда нужен именно `StringBuffer`.

**Ваш ответ:**

Разница заключается в потокобезопасности. Методы `StringBuffer` синхронизированы (synchronized), поэтому он безопасен для использования в многопоточной среде, но работает медленнее. `StringBuilder` не синхронизирован, работает быстрее, но подходит только для однопоточных задач.

**Сценарий для StringBuffer:** Несколько разных потоков (Threads) одновременно дописывают логи работы приложения в один общий строковый буфер в памяти перед тем, как скинуть их в файл.

## Вопрос 7

Что выведет этот код? Объясните, что происходит с переменной `x` в лямбде.

```java
int x = 10;
Runnable r = () -> System.out.println(x);
// x = 20;  // что будет, если раскомментировать?
r.run();
```

**Предсказанный вывод:**

```
10
```

**Объяснение:**

Лямбда-выражения могут использовать (захватывать) локальные переменные из внешней области видимости, только если они являются **`final` или "effectively final"** (фактически финальными — то есть их значение не меняется после инициализации).

Если раскомментировать строку `x = 20;`, переменная `x` перестанет быть effectively final, и код **не скомпилируется** с ошибкой: *local variables referenced from a lambda expression must be final or effectively final*.

## Вопрос 8

Напишите пример лямбда-выражения, которое нельзя заменить на ссылку на метод. Объясните почему.

**Ваш пример и объяснение:**
```java
Function<String, String> addExclamation = s -> s + "!";
```
**Объяснение:** 

Ссылку на метод (`Class::method`) можно использовать только тогда, когда лямбда просто вызывает существующий метод, передавая ему аргументы "как есть". В данном примере мы выполняем дополнительную операцию — конкатенацию строки (добавляем символ `!`). Так как мы модифицируем аргумент/создаем новую логику, заменить это на Method Reference невозможно.

## Вопрос 9

Чем `EnumMap` быстрее `HashMap` для ключей-enum? Опишите внутреннюю реализацию в одном предложении.

**Ваш ответ:**

`EnumMap` работает намного быстрее, потому что ему не нужно вычислять хэш-коды и разрешать коллизии; под капотом он реализован как обычный плоский массив (`Object[]`), где индексом элемента выступает порядковый номер enum-константы (`ordinal()`).

## Вопрос 10

Дан `record Point(int x, int y) {}`. Что сгенерирует компилятор автоматически? Можно ли добавить в record изменяемое поле? Можно ли наследоваться от record?

**Ваш ответ:**
1. **Что компилятор сгенерирует:** `private final` поля `x` и `y`, конструктор со всеми аргументами, методы-геттеры, которые называются так же, как поля (`x()` и `y()`), а также корректные реализации методов `equals()`, `hashCode()` и `toString()`.
2. **Можно ли добавить в record изменяемое поле:** Нет. Все компоненты состояния, объявленные в заголовке record, неявно становятся `final`. (Хотя можно хранить ссылку на изменяемый объект, например `List`, саму ссылку переназначить нельзя).
3. **Можно ли наследоваться:** Нет. `record` неявно является `final` классом (от него нельзя наследоваться) и сам он неявно наследует `java.lang.Record` (поэтому не может наследовать другие классы).
